## 2. 要求仕様書（顧客ヒアリング結果）

### 2.1 顧客の要望（ヒアリングメモ）

> ここに、顧客（講師）へのヒアリングで得た要望を箇条書きで記録してください。

* C言語のみで実装
* 借りる際に支払い（例：7泊8日100円　延滞は返却時に支払う）
* 返却は営業時間内（メンテナンスは営業時間外）
* 年中無休

* CSVによる書籍ID管理（DBは使用しない）
* 返却方法の指定なし
* 貸出ステータス（貸出可、貸出不可、貸出中、延滞）
* 返却時破損した場合は、人間が受け付けて除外
* 書籍数は膨大(一旦10^5,10^6)
* 延長機能はなし　延滞扱い
* 貸出上限はなし

* 利用者IDの発行は紙またはアプリ
* 操作ログのCSV形式の提案
* 利用者IDは（氏名、電話、メール、住所、（支払い方法））
* 利用者IDの追加と削除
* 延滞者の管理

### 2.2 機能要件

| 機能ID | 機能名 | 概要 |
| :---: | :--- | :--- |
| F01 | 書籍ID管理 | 書籍IDを入力し、書籍の概要および貸出状態を表示 |

## 4. 詳細設計（アルゴリズム）

### 4.1 メインループ

> メニュー表示→入力受付→処理分岐→ループの流れを記述してください。
> フローチャートまたは疑似コード（箇条書き）で記述してください。

```
1. メインメニューを表示する
2. ユーザーの入力を受け取る
3. 入力に応じて分岐する
   - 1の場合: 貸出
   - 2の場合: 返却
   - 3の場合: 書籍検索
   - 4の場合: 貸出履歴
   - それ以外の場合: 再度入力を求める
4. メインメニューに戻る（1へ）
```

### 4.2 個別機能のアルゴリズム

> 各機能（購入処理、ログ保存、ログ表示など）の処理フローを記述してください。

#### 機能1: 貸出

```
書籍IDを入力を受け取る(scanf)→書籍概要,貸出状態表示(printf)
```

#### 機能2: 返却

```
1.ユーザーのIDを入力させる
2.現在の貸出中の一覧を表示する（書籍ID、書籍タイトル、巻数、貸出日、返却期限日）
同画面内に（１つ前に戻る:0）（メインメニューへ:9）（返却する:1）を表示
3.（返却する:1）入力した場合、裏で日数超過分の金額計算を行う。超過していた場合、決済画面へ移る。
"日付超過分の金額を支払ってください"（戻るボタンは表示しない）
4.金額確認処理をして"完了"

3.（返却する:1）入力した場合、裏で日数超過分の金額計算を行う。超過していなかった場合、"完了"で処理終了（break）。

・現在貸出中の書籍情報は会員IDに登録し、会員情報から持ってくる。
・返却が完了したら履歴を上書き保存する。(貸出中から返却済みへ)
・返却延滞料金は1日経過ごとに蓄積されていく。
　↑延滞料金１日何円にする？５０円？１００円？
```

#### 機能3: 書籍検索

```
ホームメニューから「３」を入力し、書籍検索画面へ移動

１，ユーザーに書籍情報　（書籍ID）　を入力してもらうscanf　

２，書籍情報出力する

３，終了をしてホームに戻る
```

#### 機能4: 貸出履歴

```
貸出履歴閲覧画面
1. 利用者IDの入力（前画面に戻る入力を別途受け付ける）
2. 構造体Userinfoの.now_books（現在借りている書籍）と.past_books（過去借りた書籍）の書籍IDを取得（本の貸出状態の表示追加？）
3. 書籍IDから書籍情報を取得
4. メインメニューに自動に戻る

構造体配列から該当する利用者の検索は、線形探索。実装可能であれば、二部探索
書籍IDから書籍情報の取得はHashMapを用いる
```
### 4.3 エラーハンドリング（想定される異常系）

> ユーザーが「想定外の操作」をした場合にどうするか検討してください。

| 異常系シナリオ | 対処方針 |
| :--- | :--- |
| 数字以外の文字が入力された場合 | 再度入力を求める |
| 金額が不足している場合 | 「料金が不足しています」と表示し、料金の追加を求める |
